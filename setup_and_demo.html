BioX-A Biostimulant — AI/ML POC

End-to-end proof-of-concept with an experiments management layer (notebook + Streamlit + Postgres + MLflow)

Synthetic POC evidence only — not proof of product efficacy

Overview

This project is a self-contained POC demonstrating how an AI/ML platform can augment BioX's R&D workflow for the seaweed biostimulant BioX-A on tomato in tropical Asia. It integrates synthetic field, satellite, soil, weather, treatment, and yield data; recovers a known synthetic treatment effect; evaluates model quality; and generates field-level recommendations with ROI and confidence.

Prerequisites

  • Python 3.9 or newer
  • A local PostgreSQL server
  • ~2 GB free disk (for MLflow artifacts and Postgres data)

Installation

1

Install & start PostgreSQL

macOS (Homebrew):

brew install postgresql@16
brew services start postgresql@16
createdb biox_poc

Ubuntu / Debian:

sudo apt-get install postgresql
sudo service postgresql start
sudo -u postgres createdb biox_poc

Docker (alternative):

docker run --name biox-pg -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=biox_poc -p 5432:5432 -d postgres:16
2

Create a Python environment & install packages

python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

pip install pandas numpy sqlalchemy psycopg2-binary \
    scikit-learn xgboost lightgbm shap statsmodels \
    matplotlib seaborn plotly mlflow streamlit econml jupyter
econml is optional. If it is not installed, the uplift model automatically falls back to a statsmodels OLS causal estimator.
3

Configure the database connection

Defaults (used by both the notebook and dashboard):

host=localhost  port=5432  dbname=biox_poc  user=postgres  password=postgres

Override with environment variables if your setup differs:

export BIOX_PG_HOST=localhost
export BIOX_PG_PORT=5432
export BIOX_PG_DB=biox_poc
export BIOX_PG_USER=postgres
export BIOX_PG_PASSWORD=postgres

Or edit the PG_CONFIG dictionary in the notebook's Config cell directly.

Running the POC

4

Run the notebook (must run first)

Open biox_e2e_poc.ipynb in Jupyter / VS Code and choose Kernel → Restart & Run All. It is idempotent (safe to re-run). This:

  • creates the 7 agronomic tables and generates ~500 synthetic fields
  • engineers features into feature_store
  • trains the three models (yield, uplift, recommendation)
  • creates the biox_experiments schema, runs genuine experiments + evaluations, seeds mock history, and writes field_recommendations
  • logs to a local ./mlruns folder (no MLflow server required)
5

Launch the dashboard

streamlit run biox_experiments_dashboard.py
The dashboard is read-only. It requires Postgres running and the notebook to have been run at least once. If the schema is empty it shows a friendly setup message instead of erroring.
6

(Optional) Browse MLflow

mlflow ui --backend-store-uri ./mlruns

Then open http://localhost:5000 to see parent runs (trained models) and nested child runs (evaluations).

"Notice the naive treated-minus-control difference. A naive average can be misleading because treatment isn't perfectly balanced across stress conditions. This is exactly why we need a causal model, not just a group mean."

4:30 — Feature engineering (notebook §5)
"We engineer biostimulant-relevant features: NDRE/NDWI deltas (difference-in-differences), a moderate-water-stress window flag, rainfall anomaly, and treatment-by-stress interactions. These are persisted to a feature_store table so every model and every experiment reads the same governed features."

6:00 — Three models (notebook §6)
"First, a yield model — linear baseline plus Random Forest and XGBoost — validated with leave-site-out cross-validation, because in the field you deploy to sites you've never trained on. Second, a causal uplift model that isolates BioX-A's effect from everything else. Third, a deterministic recommendation engine — no black box — that only says 'apply' when uplift is positive, ROI clears a threshold, confidence is acceptable, the crop stage is right, and the soil isn't too saline."
DO: Point at the leave-site-out RMSE / R² and the uplift-by-segment table.
8:00 — The money slide: ground-truth recovery (notebook §11)
"Here's the proof point. The true synthetic uplift under moderate stress was +0.45 t/ha. The model estimated it within a few hundredths of a tonne — and, just as important, it correctly found near-zero effect under no stress and no benefit under severe drought or salinity. The platform recovered the whole response pattern, including where the product should NOT be used."
DO: Show the ground-truth recovery table sorted by |bias|. This is the single most important moment of the demo.
9:30 — Recommendations & ROI (notebook §12)
"This produces the deliverable BioX's agronomists actually want: per field, a predicted yield, an uplift percentage, an ROI multiple, a confidence level, an apply / don't-apply call, and the supporting evidence — including satellite response. And the language is disciplined: 'based on synthetic POC evidence,' never 'proven to increase yield.'"

Anticipated questions

QuestionAnswer
"Is this proving BioX-A works?"No — the data is synthetic. It proves the platform recovers a known effect, so it will surface the real effect from real data.
"Why leave-site-out validation?"Field deployment targets new sites. Random splits leak site information and overstate accuracy.
"Why a separate uplift model?"A yield model predicts outcomes; only a causal/uplift model isolates the treatment's incremental effect vs. control.
"How does this scale?"MLflow + indexed Postgres handle hundreds of experiments and evaluations; add pagination for very large histories.
"Can we add more crops/regions?"Yes — each is a new Project in the same schema; the hierarchy supports many projects.

Fully local, no cloud dependencies. The synthetic generator feeds Postgres; the notebook reads from Postgres to engineer features and train models; the experiments layer tracks every run in MLflow and mirrors metadata back into Postgres; the Streamlit dashboard reads both.

Entity relationships

Agronomic model (public)

All tables hang off fields via field_id. weather and satellite_indices are one-to-many (a pre and a post row per field).

fields (field_id PK) ├─ soil (field_id PK/FK) 1:1 ├─ treatments (field_id PK/FK) 1:1 → formulations(product_id) ├─ weather (field_id FK, period) 1:2 (pre/post) ├─ satellite_indices (field_id FK, period) 1:2 (pre/post) ├─ outcomes (field_id PK/FK) 1:1 └─ ground_truth (field_id) ← evaluation only, not a feature

Experiments model (biox_experiments)

projects (project_id PK) └─ experiments (project_id FK) └─ experiment_runs (experiment_id FK) ← a trained model ├─ run_parameters (run_id FK) ├─ run_artifacts (run_id FK) └─ evaluations (run_id FK) ← scoring under a scenario ├─ evaluation_metrics (evaluation_id FK) ← metrics live here ├─ evaluation_artifacts (evaluation_id FK) ├─ recommendation_rules (evaluation_id FK) └─ field_subsets (evaluation_id FK) treatment_configs (config_id PK) ← referenced by evaluations experiment_comparisons ← saved sets of evaluation_ids / run_ids VIEWS: v_evaluation_summary, v_run_summary (pivoted metrics for the dashboard)
Key idea: metrics attach to evaluations, not runs. One trained model (Run) can be scored under many scenarios (Evaluations) without retraining — this is what lets the platform scale to hundreds of evaluations.

Models

1. Yield prediction

Predicts yield_t_ha from soil, weather, satellite, treatment, and crop-stage features.

AspectDetail
AlgorithmsLinear Regression (baseline), Random Forest, XGBoost, LightGBM
ValidationLeave-site-out CV (each of 8 sites held out once) + optional random 80/20 grouped by site
ExplainabilitySHAP summary + feature importances (logged as artifacts)
MetricsRMSE, MAE, R² (per fold and aggregate)

2. Causal uplift

Estimates BioX-A's incremental effect vs. control — the conditional average treatment effect (CATE) per field.

AspectDetail
Methodsstatsmodels OLS causal (treatment × stress interactions + DiD covariates); optional EconML CausalForestDML
FallbackIf econml is missing, causal_forest runs fall back to OLS automatically
SegmentsCATE reported by stress regime and compared to known ground truth
MetricsATE, treatment-effect error (MAE), uplift_bias = estimated − true moderate-stress uplift

3. Recommendation engine (deterministic)

Not a black box — a transparent rule set. Recommend BioX-A only if all conditions hold:

predicted_uplift  > 0
ROI               >= roi_threshold        (default 1.5x)
uplift_ci_low     >= confidence_lower_bound (default -0.05 t/ha)
crop_stage        in {flowering, fruit_set}
soil_ec           <= max_soil_ec           (default 4.0 dS/m)

ROI = (pred_uplift_t_ha × crop_price_usd_t) / total_cost_usd, per hectare.
